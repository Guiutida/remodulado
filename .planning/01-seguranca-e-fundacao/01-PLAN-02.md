# Plano 2: Hash de Senhas com bcrypt

**Fase:** 1 — Segurança e Fundação  
**Objetivo:** Eliminar VULN-1 — substituir senhas em texto plano por hashes bcrypt no cadastro, login e atualização de perfil, e migrar senhas existentes no banco.  
**Requisitos cobertos:** AUTH-01  
**Dependências:** PLAN-01 deve estar completo (bcrypt instalado)  
**Estimativa:** 3 tarefas, ~1,5 horas

---

## Contexto

**VULN-1 confirmada:** O `server.js` armazena e compara senhas em texto plano em três pontos:

1. **Linha 170–172 — INSERT no cadastro:** `[nome, email, senha, perfil]` — a senha digitada vai direto para o banco.
2. **Linha 201–208 — SELECT no login:** `WHERE email = ? AND senha = ? AND perfil = ?` — comparação feita pelo MySQL, sem hash.
3. **Linhas 248–250 — UPDATE no perfil:** `valores.push(senha)` — nova senha salva sem hash. Este ponto é frequentemente esquecido e quebraria o login após o usuário alterar a senha.

O campo `senha VARCHAR(255)` já suporta os 60 caracteres de um hash bcrypt — nenhuma migração de schema é necessária.

**O que NÃO muda neste plano:** nenhum middleware JWT ainda, nenhuma refatoração de módulos. As três correções acontecem diretamente no `backend\server.js` monolítico atual. O PLAN-03 cuida de JWT; o PLAN-05 cuida da refatoração.

**Custo bcrypt:** `SALT_ROUNDS = 12` — boa prática para hardware 2025 (≈250ms por hash). Cadastro e troca de senha ocorrem raramente; o custo é aceitável.

## Tarefas

### Tarefa 1: Corrigir cadastro e login para usar bcrypt

**Arquivo(s):** `backend\server.js`  
**Ação:** editar — linhas 158–214

**1a — Adicionar `require` do bcrypt no topo do arquivo:**

Após a linha 5 (`const banco = require("./db");`), inserir:
```js
const bcrypt = require("bcrypt");
const SALT_ROUNDS = 12;
```

**1b — Corrigir o cadastro (linhas 169–173):**

Localizar o bloco do INSERT em `/api/cadastro`. Antes era:
```js
// server.js linhas 170–172 — INSEGURO (remover):
const [resultado] = await banco.query(
    "INSERT INTO usuarios (nome, email, senha, perfil) VALUES (?, ?, ?, ?)",
    [nome, email, senha, perfil]
);
```

Substituir por:
```js
// NOVO: hash antes do INSERT
const hash = await bcrypt.hash(senha, SALT_ROUNDS);
const [resultado] = await banco.query(
    "INSERT INTO usuarios (nome, email, senha, perfil) VALUES (?, ?, ?, ?)",
    [nome, email, hash, perfil]
);
```

**1c — Corrigir o login (linhas 200–210):**

Localizar o handler `/api/login`. Antes era:
```js
// server.js linhas 201–208 — INSEGURO (remover):
const [usuarios] = await banco.query(
    "SELECT id, nome, email, perfil FROM usuarios WHERE email = ? AND senha = ? AND perfil = ? LIMIT 1",
    [email, senha, perfil]
);

if (!usuarios.length) {
    return res.status(401).json({ status: "erro", message: "E-mail, senha ou perfil inválidos." });
}

res.json({ status: "ok", usuario: usuarios[0] });
```

Substituir por:
```js
// NOVO: buscar por email+perfil, comparar hash em JS
const [usuarios] = await banco.query(
    "SELECT id, nome, email, perfil, senha FROM usuarios WHERE email = ? AND perfil = ? LIMIT 1",
    [email, perfil]
);

if (!usuarios.length) {
    return res.status(401).json({ status: "erro", message: "E-mail, senha ou perfil inválidos." });
}

const usuario = usuarios[0];
const senhaValida = await bcrypt.compare(senha, usuario.senha);

if (!senhaValida) {
    return res.status(401).json({ status: "erro", message: "E-mail, senha ou perfil inválidos." });
}

// Não expor o campo senha na resposta
const { senha: _descartada, ...usuarioSeguro } = usuario;
res.json({ status: "ok", usuario: usuarioSeguro });
```

> **Por que mudar o SELECT?** A comparação `senha = ?` no SQL era timing-unsafe (MySQL retorna imediatamente se o email não existe). O novo padrão busca por email+perfil e delega a comparação ao `bcrypt.compare()`, que é timing-safe. A mensagem de erro propositalmente não diferencia "usuário não encontrado" de "senha errada" — isso evita enumeração de usuários.

> **Por que incluir `senha` no SELECT?** O `bcrypt.compare()` precisa do hash armazenado para comparar. O campo é desestruturado fora da resposta (`_descartada`) — nunca enviado ao browser.

**Verificação:**
```bash
curl -s -X POST http://localhost:3000/api/cadastro \
  -H "Content-Type: application/json" \
  -d "{\"nome\":\"Teste\",\"email\":\"teste@test.com\",\"senha\":\"abc123\",\"perfil\":\"aluno\"}" | node -e "const d=require('fs').readFileSync('/dev/stdin','utf8'); const r=JSON.parse(d); console.log(r.status === 'ok' ? 'PASS' : 'FAIL', r)"
```

**Arquivo(s):** `backend\server.js`

Localizar o bloco `if (senha)` dentro do handler `PUT /api/usuarios/:id`. Antes era:
```js
// server.js linhas 248–250 — INSEGURO (remover):
if (senha) {
    campos.push("senha = ?");

```

Substituir por:

> **Atenção:** como o handler usa `async (req, res)` e o `bcrypt.hash()` é assíncrono, confirmar que o handler já está marcado como `async` (está, desde a linha 233 original).

### Tarefa 3: Criar e executar script de migração de senhas existentes

**Arquivo(s):** `backend\scripts\migrar-senhas.js`  
**Ação:** criar + executar uma vez

Criar o diretório `backend\scripts\` e o arquivo `migrar-senhas.js`:

```js
// backend/scripts/migrar-senhas.js
// Executar UMA ÚNICA VEZ para migrar senhas em texto plano para bcrypt.
// Heurística: bcrypt hashes têm exatamente 60 chars; senhas curtas são texto plano.

require("dotenv").config({ path: require("path").join(__dirname, "../.env") });

const banco = require("../db");
const bcrypt = require("bcrypt");

const SALT_ROUNDS = 12;

async function migrarSenhas() {
    const [usuarios] = await banco.query(
        "SELECT id, senha FROM usuarios WHERE LENGTH(senha) < 60"
    );

    console.log(`Encontrados ${usuarios.length} usuário(s) com senha em texto plano.`);

    if (!usuarios.length) {
        console.log("Nenhuma migração necessária.");
        process.exit(0);
    }

    for (const u of usuarios) {
        const hash = await bcrypt.hash(u.senha, SALT_ROUNDS);
        await banco.query("UPDATE usuarios SET senha = ? WHERE id = ?", [hash, u.id]);
        console.log(`  Usuário ID ${u.id} migrado.`);
    }

    console.log("Migração concluída com sucesso.");
    process.exit(0);
}

migrarSenhas().catch((erro) => {
    console.error("Erro na migração:", erro.message);
    process.exit(1);
});
```

Executar o script:
```bash
cd backend && node scripts\migrar-senhas.js
```

> **Por que `LENGTH(senha) < 60`?** Um hash bcrypt tem sempre exatamente 60 caracteres no formato `$2b$12$...`. Qualquer valor mais curto no campo `senha` é texto plano. Esta heurística é segura para o banco de desenvolvimento (dados fictícios conhecidos).

> **Idempotente:** o script pode ser executado novamente sem dano — a segunda execução encontrará 0 usuários com `LENGTH(senha) < 60` e terminará sem alterar nada.

**Verificação:**
```bash
cd backend && node scripts\migrar-senhas.js
# Deve imprimir "Nenhuma migração necessária." na segunda execução.
```

Verificar no banco:
```sql
SELECT id, nome, LEFT(senha, 7) AS prefixo_senha FROM usuarios;
-- Todos os prefixos devem ser "$2b$12" (hash bcrypt)
```

- [ ] Script `migrar-senhas.js` executa sem erro e reporta 0 usuários na segunda execução
- [ ] Nenhum registro em `usuarios` tem `LENGTH(senha) < 60` após a migração
- [ ] Servidor inicia sem erros após as edições

## Ameaças e Mitigações

| Ameaça | Mitigação |
|--------|-----------|
| Campo `senha` exposto na resposta da API | Desestruturação `{ senha: _descartada, ...usuarioSeguro }` antes do `res.json()` — campo nunca sai do servidor |
| Timing attack no login (resposta mais rápida quando usuário não existe) | `bcrypt.compare()` é timing-safe; mensagem de erro genérica (não diferencia usuário inexistente de senha errada) |
| Script de migração executado em produção acidentalmente | Script usa `process.exit(0)` ao final — não é um servidor. Documentar no README que deve ser executado apenas em desenvolvimento |
| `bcryptjs` instalado por engano no lugar de `bcrypt` | PLAN-01 instalou `bcrypt` com bindings C++. Confirmar com `node -e "const b = require('bcrypt'); console.log(b.getRounds('test'))"` — se lançar exceção é o pacote errado |
