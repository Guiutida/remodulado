**Objetivo:** Aplicar o middleware `autenticar` em todas as rotas protegidas, eliminar IDOR nas 6 rotas vulneráveis, remover `garantirAtividadeFuncoes()` e `garantirTabelaPreferencias()` do fluxo de produção.  
**Requisitos cobertos:** AUTH-02, AUTH-06, INFRA-02

## Contexto

**VULN-2 — Sem JWT:** O middleware `autenticar` foi criado no PLAN-03 mas ainda não está em nenhuma rota. Este plano faz a ligação.

**VULN-3 — IDOR em 6 rotas (linhas 216, 233, 276, 300, 341, 369):**

| Rota | Linha | Problema |
|------|-------|---------|
| `GET /api/usuarios/:id` | 216 | Qualquer autenticado lê dados de qualquer usuário |
| `PUT /api/usuarios/:id` | 233 | Qualquer autenticado altera dados de qualquer usuário |
| `GET /api/usuarios/:id/preferencias` | 276 | Qualquer autenticado lê preferências de qualquer usuário |
| `PUT /api/usuarios/:id/preferencias` | 300 | Qualquer autenticado altera preferências de qualquer usuário |
| `GET /api/alunos/:id/progresso/funcoes` | 341 | Qualquer autenticado lê progresso de qualquer aluno |
| `POST /api/alunos/:id/progresso/funcoes` | 369 | Qualquer autenticado grava progresso de qualquer aluno |

**Regra de acesso:** aluno só acessa os próprios dados (`req.usuario.id === parseInt(req.params.id, 10)`). Professor pode acessar dados de qualquer usuário (necessário para gestão pedagógica).

**VULN-5 — Seed de produção:** `garantirAtividadeFuncoes()` é chamada nas linhas 343 e 371 a cada request de progresso. Ela cria usuário, turma, atividade e vínculo fictícios se não existirem — comportamento de bootstrap de banco, não de produção. A função inteira (linhas 34–94) deve ser removida do `server.js`. A rota de progresso passa a buscar entregas diretamente sem a função auxiliar.

**`garantirTabelaPreferencias()` (linhas 96–119):** Executa `CREATE TABLE IF NOT EXISTS` em toda requisição de preferências. A tabela já existe em `database/schema.sql`. Remover a chamada e a função — mas **preservar `garantirPreferencias()`** (que faz INSERT IGNORE de preferências padrão para o usuário). Apenas a criação dinâmica de tabela é removida.

**O que NÃO muda neste plano:** estrutura de módulos (server.js ainda monolítico), frontend — isso é PLAN-05.

## Critérios de Aceite

- [ ] `GET /api/usuarios/1` sem token retorna HTTP 401 `{ status: "erro", message: "Não autenticado." }`
- [ ] `GET /api/usuarios/1` com token válido de usuário ID 1 retorna HTTP 200 com dados do usuário
- [ ] `GET /api/usuarios/2` com token válido de usuário ID 1 (aluno) retorna HTTP 403 `{ status: "erro", message: "Acesso negado." }`
- [ ] `GET /api/usuarios/2` com token válido de professor retorna HTTP 200 (professor pode ver qualquer usuário)
- [ ] `GET /api/alunos/999/progresso/funcoes` com token de usuário ID 1 retorna HTTP 403
- [ ] A função `garantirAtividadeFuncoes` não existe mais no `server.js` (verificar com `grep "garantirAtividadeFuncoes" backend\server.js` — deve retornar vazio)
- [ ] A função `garantirTabelaPreferencias` não existe mais no `server.js`
- [ ] `GET /api/alunos/:id/progresso/funcoes` com token válido do próprio aluno retorna HTTP 200 (sem criar usuários/turmas fictícias)
- [ ] Banco de dados não contém registro `ana@duopratic.local` criado por chamadas de produção após este plano
- [ ] `/api/health`, `/api/cadastro`, `/api/login` continuam acessíveis sem token

## Ameaças e Mitigações

| Ameaça | Mitigação |
|--------|-----------|
| IDOR por comparação de tipo errado (string vs int) | `parseInt(req.params.id, 10)` garante comparação numérica com `req.usuario.id` do JWT |
| Enumeração de usuários via IDOR (403 revela que o usuário existe) | 403 é aceitável — o usuário já está autenticado; não é enumeração anônima |
| Dados de seed recriados em produção | `garantirAtividadeFuncoes` removida do fluxo. Script de seed movido para `backend\scripts\` apenas |
| `CREATE TABLE` em produção por cada request | `garantirTabelaPreferencias` removida; tabela existe no schema.sql |
| Professor acessando dados indevidamente | Restrição de papel implementada: professor só acessa via `/usuarios/:id`, não via `/alunos/:id/progresso` |
