## Contexto

**VULN-2 — Sem middleware JWT:** Nenhuma rota atual exige autenticação. Qualquer `fetch` retorna dados.

**VULN-4 — CORS wildcard:** O middleware manual nas linhas 22–32 do `server.js` tem dois problemas cumulativos:
1. `Access-Control-Allow-Origin: *` — aceita qualquer origem.
2. `Access-Control-Allow-Headers: Content-Type` — não inclui `Authorization`. Isso causará falha de preflight CORS imediatamente ao adicionar o cabeçalho Bearer em qualquer `fetch`. **O CORS deve ser corrigido neste plano, antes do PLAN-04 aplicar o middleware JWT nas rotas** — do contrário o frontend vai quebrar por CORS antes de chegar no JWT.

**Ordem crítica das tarefas neste plano:**
1. Criar `backend\middleware\auth.js` (middleware JWT — ainda não aplicado a rotas)
2. Adicionar emissão de token JWT no handler `/api/login` do `server.js`
3. Substituir CORS manual + adicionar helmet e rate limiting no `server.js`

O PLAN-04 aplica o middleware às rotas. O PLAN-05 refatora os módulos.

## Tarefas

### Tarefa 1: Criar backend\middleware\auth.js

**Arquivo(s):** `backend\middleware\auth.js` *(novo)*  
**Ação:** criar

Criar o diretório `backend\middleware\` e o arquivo `auth.js`:

```js
// backend/middleware/auth.js
// Middleware JWT — verifica token Bearer e popula req.usuario = { id, perfil }
// Uso nas rotas: router.get("/rota", autenticar, handler)

const jwt = require("jsonwebtoken");

function autenticar(req, res, next) {
    const cabecalho = req.headers.authorization;

    if (!cabecalho || !cabecalho.startsWith("Bearer ")) {
        return res.status(401).json({ status: "erro", message: "Não autenticado." });
    }

    const token = cabecalho.slice(7); // remove "Bearer "

    try {
        req.usuario = jwt.verify(token, process.env.JWT_SECRET);
        // req.usuario agora contém: { id, perfil, iat, exp }
        next();
    } catch {
        // jwt.verify lança JsonWebTokenError se inválido, TokenExpiredError se expirado
        res.status(401).json({ status: "erro", message: "Token inválido ou expirado." });
    }
}

module.exports = { autenticar };
```

> **Padrão do projeto:** `module.exports = { autenticar }` (objeto, não função nua) — facilita adicionar `verificarProprioUsuario` no mesmo módulo no PLAN-04 se necessário. Controllers importam com `const { autenticar } = require("../middleware/auth")`.

> **Não usar `req.headers["authorization"]` com `split(" ")[1]`:** o padrão `cabecalho.slice(7)` é mais simples e lida corretamente com tokens que contêm espaços em payloads incomuns.

> **Por que não logar o erro do jwt.verify?** Tokens inválidos são frequentes em produção (expirados, adulterados). Logar cada um criaria ruído. O status 401 com mensagem genérica é suficiente.

**Verificação:**
```bash
node -e "
const { autenticar } = require('./backend/middleware/auth');
console.log(typeof autenticar === 'function' ? 'PASS: autenticar é função' : 'FAIL');
"
```

### Tarefa 2: Emitir JWT no login (server.js)

**Arquivo(s):** `backend\server.js`  
**Ação:** editar — handler POST /api/login

Adicionar `require` do jsonwebtoken logo após o `require` do bcrypt (adicionado no PLAN-02, linhas 6–7):
```js
const jwt = require("jsonwebtoken");
```

Localizar o final do handler `/api/login`, onde o PLAN-02 deixou:
```js
// Não expor o campo senha na resposta
const { senha: _descartada, ...usuarioSeguro } = usuario;
res.json({ status: "ok", usuario: usuarioSeguro });
```

Substituir por:
```js
// Emitir token JWT
const token = jwt.sign(
    { id: usuario.id, perfil: usuario.perfil },
    process.env.JWT_SECRET,
    { expiresIn: "7d" }
);

// Não expor o campo senha na resposta
const { senha: _descartada, ...usuarioSeguro } = usuario;
res.json({ status: "ok", token, usuario: usuarioSeguro });
```

> **`expiresIn: "7d"`:** 7 dias é adequado para um app educacional (não bancário). O frontend salva o token no localStorage; após 7 dias o usuário precisará fazer login novamente. Isso cobre o AUTH-04 (persistência entre recarregamentos) sem sessões eternas.

> **Payload mínimo `{ id, perfil }`:** não incluir nome, email ou outros dados no payload — eles podem mudar e o token não seria atualizado. O `req.usuario.id` é o suficiente para verificações IDOR; `req.usuario.perfil` é o suficiente para controle de acesso por papel.

**Verificação:**
```bash
curl -s -X POST http://localhost:3000/api/login \
  -H "Content-Type: application/json" \
  -d "{\"email\":\"teste@test.com\",\"senha\":\"abc123\",\"perfil\":\"aluno\"}" \
  | node -e "const d=require('fs').readFileSync('/dev/stdin','utf8'); const r=JSON.parse(d); console.log(r.token ? 'PASS: token presente' : 'FAIL: token ausente', r.status)"
```
(Usar credenciais de um usuário existente no banco de dev.)

### Tarefa 3: Substituir CORS manual por pacote cors + adicionar helmet e rate limiting

**Arquivo(s):** `backend\server.js`  
**Ação:** editar — linhas 1–32 (seção de setup do Express)

**3a — Adicionar requires no topo:**

Após os requires existentes (`express`, `path`, `banco`, `bcrypt`, `jwt`), adicionar:
```js
const cors = require("cors");
const helmet = require("helmet");
const rateLimit = require("express-rate-limit");
```

**3b — Configurar rate limiter para rotas de auth:**

Antes da linha `app.use(express.json())`, adicionar:
```js
const limitadorAuth = rateLimit({
    windowMs: 15 * 60 * 1000, // 15 minutos
    max: 20,                   // máximo 20 tentativas por IP
    standardHeaders: true,
    legacyHeaders: false,
    message: { status: "erro", message: "Muitas tentativas. Tente novamente em 15 minutos." }
});
```

**3c — Substituir o middleware CORS manual (linhas 22–32) pelo pacote cors + helmet:**

Remover completamente:
```js
// REMOVER — linhas 22–32 do server.js original:
app.use((req, res, next) => {
    res.setHeader("Access-Control-Allow-Origin", "*");
    res.setHeader("Access-Control-Allow-Headers", "Content-Type");
    res.setHeader("Access-Control-Allow-Methods", "GET, POST, PUT, OPTIONS");

    if (req.method === "OPTIONS") {
        return res.sendStatus(204);
    }

    next();
});
```

Substituir por (inserir ANTES de `app.use(express.json())`):
```js
// Segurança HTTP — helmet adiciona ~12 headers de proteção (CSP, HSTS, X-Frame-Options, etc.)
app.use(helmet());

// CORS — substituir wildcard; incluir Authorization para suportar Bearer token
app.use(cors({
    origin: process.env.CORS_ORIGIN || "http://localhost:3000",
    methods: ["GET", "POST", "PUT", "DELETE", "OPTIONS"],
    allowedHeaders: ["Content-Type", "Authorization"]
}));
```

**3d — Aplicar rate limiter nas rotas de auth:**

Localizar as linhas `app.post("/api/cadastro", ...)` e `app.post("/api/login", ...)` e adicionar o limitador como segundo argumento:
```js
app.post("/api/cadastro", limitadorAuth, async (req, res) => { ... });
app.post("/api/login",    limitadorAuth, async (req, res) => { ... });
```

> **Por que `allowedHeaders: ["Content-Type", "Authorization"]` é crítico?** O CORS manual atual só lista `Content-Type`. Quando o PLAN-04 adicionar o middleware JWT, o browser enviará um preflight `OPTIONS` pedindo permissão para o header `Authorization`. Sem ele na allowlist, o browser bloqueará a requisição antes de chegar no Express — erro de CORS no console, 0 resposta do servidor.

> **Por que `helmet()` antes do `cors()`?** Helmet adiciona headers de segurança à resposta; CORS também adiciona headers. A ordem não é crítica para funcionalidade, mas helmet antes é convencional.

> **`legacyHeaders: false` no rate limiter:** Desativa os headers `X-RateLimit-*` deprecados. Os novos headers RFC `RateLimit-*` são habilitados por `standardHeaders: true`.

**Verificação:**
```bash
# Servidor deve iniciar sem erros
node backend\server.js &
sleep 2

# Testar que CORS retorna o header correto (não wildcard)
curl -s -I -X OPTIONS http://localhost:3000/api/login \
  -H "Origin: http://localhost:3000" \
  -H "Access-Control-Request-Headers: Authorization" \
  | grep -i "access-control"

# Deve incluir:
# Access-Control-Allow-Origin: http://localhost:3000   (não *)
# Access-Control-Allow-Headers: Content-Type,Authorization

# Testar que helmet adicionou headers de segurança
curl -s -I http://localhost:3000/api/health | grep -iE "x-frame|x-content|strict-transport"
```

## Ameaças e Mitigações

| Ameaça | Mitigação |
|--------|-----------|
| `JWT_SECRET` fraco ou ausente | `process.env.JWT_SECRET` — se ausente no `.env`, `jwt.sign()` lança erro imediato no login. PLAN-01 garantiu que a variável existe. |
| Token JWT com payload sensível | Payload contém apenas `{ id, perfil }` — nenhum dado pessoal (nome, email, senha) |
| CORS liberado demais em produção | `CORS_ORIGIN` é configurável via `.env`. Em produção, trocar para a URL real do domínio |
| Brute-force de login | Rate limiter: 20 tentativas por IP em 15 minutos antes de bloquear |
| Preflight CORS bloqueando Authorization | `allowedHeaders: ["Content-Type", "Authorization"]` cobre o header Bearer |
| Clickjacking | `helmet()` adiciona `X-Frame-Options: SAMEORIGIN` automaticamente |
